Explain the ways to implement preprocessing (client-side vs. server-side)

In web applications, there are two variables the “client” and “server”. The “client” is a web browser, like Internet Explorer, Google Chrome, Firefox, etc. The “server” is a web application server at a remote location that will process web requests and send pages to the client. Web applications contain code that’s processed through client’s browser or web server. The disadvantage is that web applications have a disconnected architecture that is never live or having a consistent connection between displayed pages on a browser, web, or database. Server-side processing happens when a page is being reached providing the requests and when the pages are posted back to servers with information of saving/retrieving data, server-side processing, and user validation. A disadvantage of server-side processing is that it can cause decreased performance increasing loading times for users to wait to process and recreated.it can introduce processing overhead that can decrease performance and force the user to wait for the page to be processed and recreated. Client-Side Code are embedded scripts on a webpage that are processed through the client’s internet browser. Scripts are written in differencing languages of code such as JavaScript interacting with website’s HTML elements including textboxes, list-boxes, tables, and buttons. The benefits of solid client-side scripting provide the user with faster response times, intuitive interaction with applications, and less overhead from the web server. The upside of client coding is that you can make direct changes to elements without contacting the database. The downside of it all is that it requires more time and effort while the browser can support the scripted language.

Define and describes the mechanisms used in either LESS or SASS

Mixins define styles that is re-used throughout the stylesheet. Mixins use '@mixin at-rule', which is written "@mixin name' { ... } or '@mixin arguments' { ... }” Variables are values that are used infinite amount of times within the stylesheet. A value that begins with a $ creates a variable allowing to use the same name throughout the stylesheet instead of actual value written in : Function defines complex operations with SassScript values having the ability to re-use throughout your stylesheet. Functions allow common formulas and behaviors in readable format through abstraction by using @function at-rule, which is written @function () { ... } but a function’s name can be any Sass Identifier containing universal statements with @return at-rule indicating values as a result of function call. Operators consist of mathematical operations such as +, -, *, /, and %. Equality check would use == and !=. Greater or less than between two numbers use <,<=,>, and =. Words such as and, or, and not have Boolean behavior. In Sass, every value is considered “true” except for false and null. The +,-, and / are used as concentration strings. Nesting in Sass allows to nest CSS selectors in similar fashion as that of HTML. You can write a set of style rules inside of another instead of repeating same selectors. It provides style organization as every set of rules in relation to the sector will be in one place. It is not favored to implement nesting in deep levels causing harder maintenance.
